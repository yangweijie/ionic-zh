Contribute To Ionic

If you see an area for improvement, or have an idea for a new feature then we'd love to have your help! Additionally, please feel free to take part in any discussions which may be happening on our forum or our issues page. You can also stay up-to-date with Ionic by subscribing to our newsletter.

  1. Join the discussion on Ionic's Forum. Either start or take part in topics to help define what you're thinking.
  2. Browse existing issues for ideas or to see if a similar issue has already been submitted.
  3. Fork the Github repo.
  4. Create a new branch from master: git checkout -b issuenumber-new-awesomeness
  5. Review our pull request guidelines.
  6. Follow our coding conventions.
  7. Did you remember to write a passing test?
  8. Submit a pull request!

For issues relating to the Ionic documentation or this site, please use the ionic-site repo instead.

Submit An Issue For The Ionic Framework

If you see a way to improve any piece of code please let us know. There's no such thing as too small of an improvement. (If the issue is about our documentation please see the next section.) If you think you might have found a bug, please start by making sure it hasn't already been reported.

If the issue doesn't already exist, then create a plunkr that demonstrates the problem (try to use as little code as possible: the more minimalist, the faster we can debug it). Next, create a new issue that briefly explains the problem, and provides a bit of background as to the circumstances that triggered it. Don't forget to include the link to that plunkr you created!

Help Improve The Docs

See a typo, have a better way to describe Ionic within our documentation, or have a great tutorial? Please help make our docs awesome. Feel free to fork the ionic-site repo, add, correct or improve the content, then submit a pull request with the improved docs. Or if its something small just an issue will do. Thanks!

Pull Request Guidelines

When in doubt, keep your pull requests small. To give a PR the best chance of getting accepted, do not bundle more than one "feature" or bug fix per PR. A large "all-in-one" PR makes it very hard to accept if one of the fixes or features has issues.

It's always best to create two or more smaller PRs than one big one.

Conventions

We try to follow best practices as much as possible.

For any SCSS changes, try to keep CSS properties sorted in a reasonable order, with layout and sizing properties first, then cosmetic properties below that. See our code for examples. Feel free to run your code through CSS Comb, which will sort your CSS properties in a good order.

All JavaScript must pass JSHint, which helps detect errors and potential problems in your code.

Additionally, almost all Javascript code must have an accompanying test case. To run our test suite, install and run Karma as follows: 

$ sudo npm install -g karma
$ karma start ionic.conf.js

Any tests you write should go in the test/js folder for core JS code, or in the js/ext/FRAMEWORK/test folder for framework-specific code. Copy the format from other tests to write your own.

Indentation

Always use two spaces, no tabs. This goes for any HTML, CSS, or Javascript.
